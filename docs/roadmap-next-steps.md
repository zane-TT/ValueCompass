# ValueCompass 下一步落地路线图

目标：把 ValueCompass 从“单公司财报可视化 Demo”推进到“能判断公司赚不赚钱、业绩趋势好不好、质量是否可靠”的研究工具雏形。

## 优先主线

下一阶段不要同时做太多方向，优先完成这三件事：

1. 现金流与盈利质量分析
2. 同比 / TTM 业绩摘要
3. 自动风险提示模块

这三件事都服务同一个问题：这家公司最近几年到底赚不赚钱、业绩是在变好还是变差、利润质量是否可信。

## 任务 1：现金流与盈利质量分析

目标：

补上经营现金流、自由现金流、净现比、收现比，让系统能判断“利润是不是变成了现金”。

后端改动：

- 新增现金流数据读取函数，例如 `load_cash_flow_sheet(stock: str)`。
- 新增现金流序列构建函数，例如 `build_cash_flow_bars(df, years)`。
- 新增接口 `GET /api/cash-flow-quality?stock=600519&years=8`。
- 返回字段建议：

```json
{
  "stock": "600519",
  "title": "600519 现金流与盈利质量",
  "unit": "亿元",
  "operatingCashFlow": [{ "date": "2025-12-31", "value": 123.45 }],
  "netProfit": [{ "date": "2025-12-31", "value": 100.12 }],
  "cashToProfitRatio": [{ "date": "2025-12-31", "value": 1.23 }],
  "conclusion": "最近几年经营现金流持续为正，净现比大多高于 1，利润质量较好。"
}
```

前端改动：

- 新增一个图表卡片：`现金流与盈利质量`。
- 图表建议用柱状图 + 折线：
  - 柱：经营现金流、归母净利润
  - 线：净现比
- 在自动结论里加入现金流判断。

验收标准：

- 输入 `600519` 能看到经营现金流和净利润对比。
- 自动结论能说清楚“利润质量好 / 一般 / 有压力”。
- 如果现金流连续为负，要有明确风险提示。

## 任务 2：同比 / TTM 业绩摘要

目标：

让系统不只展示原始趋势，还能直接回答“最近几年业绩是什么情况”。

后端改动：

- 新增通用工具函数：
  - `group_year_end_points(points)`
  - `calculate_yoy(points)`
  - `calculate_ttm(points)`
- 在营收、净利润 payload 中补充：

```json
{
  "yearlySummary": [
    {
      "year": "2025",
      "revenue": 1000.12,
      "revenueYoY": 0.08,
      "netProfit": 200.34,
      "netProfitYoY": 0.12
    }
  ],
  "performanceConclusion": "最近三年收入持续增长，净利润增速快于收入，盈利能力改善。"
}
```

前端改动：

- 自动结论模块使用后端返回的 `performanceConclusion`。
- 增加一个“小表格”展示最近 3-5 年：
  - 年份
  - 营收
  - 营收同比
  - 归母净利润
  - 净利润同比

验收标准：

- 页面能明确显示最近几年营收和净利润的同比。
- 自动结论能区分：
  - 增收增利
  - 增收不增利
  - 收入下滑但利润改善
  - 收入利润双降
  - 亏损扩大 / 扭亏为盈

## 任务 3：自动风险提示模块

目标：

把系统从“展示数据”推进到“主动提醒异常”。

后端改动：

- 新增风险检测函数，例如 `detect_financial_risks(context)`。
- 初期规则建议：
  - 最新净利润为负：`公司最近报告期亏损`
  - 净利润连续下降：`盈利能力持续走弱`
  - 营收下降且利润下降：`经营规模和盈利同步承压`
  - 经营现金流为负：`利润现金含量不足`
  - 净现比低于 0.5：`利润质量偏弱`
  - 资产负债率过高：`负债压力较大`
  - 应收账款占比过高：`回款压力需要关注`
  - 存货占比过高：`库存消化压力需要关注`
- 新增接口可选：

```text
GET /api/risk-summary?stock=600519&period=&years=8
```

也可以先把风险结果合并到现有 `/api/ai-analysis` 或自动结论相关 payload。

前端改动：

- 新增模块：`风险提示`。
- 风险等级建议：
  - 高风险
  - 中风险
  - 观察项
- 每条风险必须显示触发原因，例如：

```text
经营现金流连续 2 年为负，说明利润尚未稳定转化为现金。
```

验收标准：

- 至少能对 600519、000333、300052、601919 跑出不同风险提示。
- 风险提示不能只写空泛句子，必须带数据依据。

## 暂不建议做的事

先不要做这些，容易把节奏打散：

- 登录系统
- 用户自选股持久化
- Docker 部署
- 多 Agent 架构
- 复杂预测模型
- 组合回测
- 实时行情

这些都重要，但不是当前最短路径。

## 推荐实现顺序

1. 先做后端现金流接口。
2. 再把现金流图表接到前端。
3. 再把自动结论改为后端统一生成。
4. 最后加风险提示规则。
5. 每完成一个接口，先用 `FastAPI TestClient` 验证，再跑 `npm.cmd run build`。
